# 开发逻辑学习笔记：AX58100 与 CoE

> 适合已经理解 SPI PDI 基础读写之后继续看。  
> 目标不是背协议，而是先把项目里的开发逻辑、分层关系、主循环路径看清楚。

---

## 1. 先看整体：这个工程每 5ms 在做什么

主循环在 `Core/Src/main.c`：

```c
ECAT_Init();
CoE_Init();
ESC_Watchdog_Config();

while (1)
{
    ECAT_MainTask();
    ECAT_MainTask();

    CoE_MainTask();
    CoE_MainTask();
    CoE_MainTask();

    ECAT_ProcessDataExchange();

    HAL_Delay(5);
}
```

可以先记成四件事：

```text
1. 状态机：看主站想让从站进入 INIT / PREOP / SAFEOP / OP 哪个状态
2. CoE：看主站有没有通过邮箱发 SDO 请求
3. 过程数据：如果已经 OP，就交换 SM2/SM3 的 PDO 数据
4. 等 5ms，再来一轮
```

当前工程的三层关系是：

```text
Core/main.c
  -> App/app_ethercat.c     EtherCAT 状态机 + 过程数据应用
  -> App/app_coe.c          CoE / SDO / 对象字典
       -> BSP/AX58100.c     ESC 寄存器、SM、邮箱、过程数据 RAM 访问
            -> Core/spi.c   SPI PDI 物理传输
```

你已经会 SPI，所以后面主要看：

```text
AX58100.c：怎么把 SPI 读写包装成 ESC/SM/邮箱/过程数据接口
app_coe.c：怎么把邮箱里的字节解释成 CoE/SDO/对象字典访问
```

## 2. AX58100 层：它是 ESC 驱动，不是应用协议

核心文件：

```text
Bsp/Inc/AX58100.h
Bsp/Src/AX58100.c
```

这一层的定位：

```text
AX58100 层只负责“怎么访问 ESC 硬件资源”
它不关心对象字典、不关心 SDO、不关心 LED 业务含义
```

也就是说，AX58100 层提供的是这些能力：

```text
读写 ESC 寄存器
读写 ESC RAM
读写 SyncManager 配置
读写 Mailbox 区域
读写过程数据区域
读 ESC 诊断信息
禁用/配置看门狗
```

### 2.1 基础访问：ESC_Read/Write

最底层的通用接口：

```c
HAL_StatusTypeDef ESC_ReadRegister(uint16_t addr, uint8_t *pData);
HAL_StatusTypeDef ESC_WriteRegister(uint16_t addr, uint8_t data);
HAL_StatusTypeDef ESC_ReadBlock(uint16_t addr, uint8_t *pData, uint16_t size);
HAL_StatusTypeDef ESC_WriteBlock(uint16_t addr, uint8_t *pData, uint16_t size);
```

理解方式：

```text
ESC_ReadRegister   读 1 个 ESC 地址
ESC_WriteRegister  写 1 个 ESC 地址
ESC_ReadBlock      连续读一段 ESC 地址
ESC_WriteBlock     连续写一段 ESC 地址
```

上层所有东西，包括：

```text
AL Control / AL Status
SM0~SM3 配置
FMMU 配置
邮箱数据
过程数据
```

最后都会落到这四类读写函数上。

### 2.2 SyncManager：定义 RAM 区域的用途

默认布局在 `AX58100.h`：

```text
SM0  0x1000  128B  主站 -> 从站  邮箱命令
SM1  0x1080  128B  从站 -> 主站  邮箱响应
SM2  0x1100   32B  主站 -> 从站  过程数据输出
SM3  0x1120   32B  从站 -> 主站  过程数据输入
```

代码接口：

```c
void ESC_SM_ReadConfig(uint8_t smIdx, ESC_SM_Config_t *pCfg);
void ESC_SM_Config(uint8_t smIdx, const ESC_SM_Config_t *pCfg);
```

重点：

```text
开发自测时，可以用 ESC_SM_Init() 写默认 SM 配置
实际连 TwinCAT 后，SM 配置可能由主站根据 EEPROM/ESI 写入
所以正式逻辑里，MCU 更多是读取并验证 SM，而不是强行覆盖
```

这也是为什么 `app_ethercat.c` 里切 PREOP/SAFEOP/OP 时会反复读 SM 配置。

### 2.3 邮箱底层：只搬运，不解释

邮箱用 SM0/SM1：

```text
SM0：主站写入命令，MCU 读取
SM1：MCU 写入响应，主站读取
```

AX58100 层提供：

```c
uint8_t ESC_Mbx_RxFull(void);
uint8_t ESC_Mbx_TxFull(void);
HAL_StatusTypeDef ESC_Mbx_Read(uint8_t *pBuf, uint16_t len);
HAL_StatusTypeDef ESC_Mbx_Write(uint8_t *pBuf, uint16_t len);
```

理解方式：

```text
ESC_Mbx_RxFull()
  判断 SM0 是否有主站发来的新邮箱

ESC_Mbx_Read()
  从 SM0 实际配置的地址读取邮箱数据

ESC_Mbx_TxFull()
  判断 SM1 上一次响应是否还没被主站取走

ESC_Mbx_Write()
  往 SM1 实际配置的地址写入响应
```

注意这里的边界：

```text
AX58100.c 不知道邮箱里是 CoE、FoE、SoE 还是别的协议
它只负责把邮箱字节读出来、写回去
```

### 2.4 过程数据底层：SM2/SM3

过程数据接口：

```c
void ESC_ReadOutputData(uint8_t *pBuf, uint16_t len);
void ESC_WriteInputData(uint8_t *pBuf, uint16_t len);
```

对应关系：

```text
ESC_ReadOutputData()
  从 SM2 读主站输出数据
  也就是 RxPDO，主站 -> 从站

ESC_WriteInputData()
  向 SM3 写从站输入数据
  也就是 TxPDO，从站 -> 主站
```

这里也不解释 PDO 映射对象，只负责从对应 RAM 区域读写字节。

## 3. EtherCAT 状态机：决定什么时候能通信、什么时候能交换过程数据

核心文件：

```text
App/Src/app_ethercat.c
App/Inc/app_ethercat.h
```

状态定义：

```text
INIT    = 0x01
PREOP   = 0x02
BOOT    = 0x03
SAFEOP  = 0x04
OP      = 0x08
```

核心入口：

```c
uint8_t ECAT_MainTask(void);
```

逻辑：

```text
1. 读 AL Control 0x0120
2. 解析主站请求的目标状态
3. 判断状态跳转是否合法
4. 在关键跳转点检查 SM/FMMU 配置
5. 写 AL Status 0x0130
6. 如果失败，写 AL Status Code 0x0134
```

### 3.1 状态跳转检查

合法跳转大概是：

```text
INIT -> PREOP
PREOP -> SAFEOP
SAFEOP -> OP
OP -> SAFEOP
SAFEOP -> PREOP
任意状态 -> INIT
```

对应代码在 `IsValidTransition()`。

### 3.2 配置检查

状态机不只是改状态，它还会在进入某些状态前检查配置：

```text
PREOP -> SAFEOP
  检查 SM0/SM1 邮箱配置
  检查 SM2/SM3 过程数据配置

SAFEOP -> OP
  检查 SM2/SM3 过程数据配置
  检查 FMMU 映射
```

如果检查失败：

```text
AL Status 保持在旧状态，并置 Error bit
AL Status Code 写对应错误码
TwinCAT 看到后会显示状态切换失败原因
```

### 3.3 OP 状态下的过程数据交换

入口：

```c
void ECAT_ProcessDataExchange(void);
```

只有当前状态是 OP 才执行：

```text
如果不是 OP：
  直接 return

如果是 OP：
  1. 从 SM2 读主站输出
  2. 更新 g_testCounter
  3. 用 g_testCounter 控制 LED
  4. 读取本地输入
  5. 更新 g_testStatus
  6. 把 g_testStatus 写到 SM3 给主站
```

这部分可以看成当前工程最小应用：

```text
主站写 0x2000 / RxPDO -> 控制 LED
从站采集输入 -> 0x2001 / TxPDO -> 主站读取
```

## 4. CoE 层：解释邮箱里的 SDO 和对象字典

核心文件：

```text
App/Src/app_coe.c
App/Inc/app_coe.h
```

这一层的定位：

```text
CoE 层不直接碰 SPI
CoE 层不直接配置 SM
CoE 层只通过 ESC_Mbx_*() 收发邮箱包
然后解析 CoE / SDO / 对象字典
```

主入口：

```c
uint8_t CoE_MainTask(void);
```

运行流程：

```text
1. ESC_Mbx_RxFull()
   看 SM0 是否有主站发来的邮箱

2. ESC_Mbx_Read()
   从 SM0 读出完整邮箱包

3. 解析 Mailbox Header
   判断 typeCounter 低 4 位是不是 MBX_TYPE_COE

4. 解析 CoE Header
   判断 service 是 SDO Request 还是 SDO Info

5. 分发处理：
   SDO Upload
   SDO Download
   SDO Info

6. 等 SM1 可写

7. ESC_Mbx_Write()
   把响应写回 SM1
```

### 4.1 Mailbox Header / CoE Header / SDO Header

`app_coe.h` 里定义了三层头：

```text
Mailbox Header  6B
CoE Header      2B
SDO Header      8B
```

可以记成：

```text
Mailbox Header
  这是什么邮箱类型？CoE/FoE/SoE？

CoE Header
  这是 CoE 里的什么服务？SDO Request? SDO Info?

SDO Header
  要访问哪个对象？index/subindex 是多少？读还是写？
```

### 4.2 对象字典

对象字典在 `app_coe.c` 的 `g_objectDict[]`。

当前主要对象：

```text
0x1000       Device Type
0x1008       Device Name
0x1009       Hardware Version
0x100A       Software Version
0x1018:01    Vendor ID
0x1018:02    Product Code
0x1018:03    Revision Number
0x1018:04    Serial Number

0x1C00       Sync Manager Communication Type
0x1600       RxPDO Mapping
0x1A00       TxPDO Mapping
0x1C12       RxPDO Assign
0x1C13       TxPDO Assign

0x2000       LED Output Mask
0x2001       Digital Input Mask
```

每个对象字典条目本质上是：

```c
typedef struct {
    uint16_t index;
    uint8_t  subindex;
    uint8_t  dataType;
    uint8_t  access;
    uint8_t  reserved;
    uint16_t dataLen;
    void    *pData;
} OD_Entry_t;
```

也就是：

```text
对象编号 + 子编号 + 数据类型 + 权限 + 长度 + 指向真实变量的指针
```

所以 TwinCAT 读 `0x1008` 时，代码就是：

```text
找 g_objectDict 里 index=0x1008/subindex=0 的条目
拿到 pData 指向的字符串 "AX58100_Test"
打包成 SDO Upload Response
通过 SM1 回给主站
```

### 4.3 SDO Upload：主站读对象

入口：

```c
static void SDO_HandleUpload(uint16_t index, uint8_t subindex);
```

典型流程：

```text
1. OD_Find(index, subindex)
2. 如果不存在，回 SDO Abort
3. 如果是只写对象，回 SDO Abort
4. 如果长度 <= 4 字节，走 expedited upload
5. 如果长度 > 4 字节，走 normal upload，后续分段发送
6. 写 SM1 响应
```

例子：

```text
主站读 0x1018:01 Vendor ID
  -> 找到 g_vendorID
  -> 长度 4 字节
  -> 直接 expedited response

主站读 0x1008 Device Name
  -> 字符串长度大于 4
  -> normal upload + segment response
```

### 4.4 SDO Download：主站写对象

入口：

```c
static void SDO_HandleDownload(SDO_Header_t *pSDO);
```

典型流程：

```text
1. OD_Find(index, subindex)
2. 如果不存在，Abort
3. 如果是只读对象，Abort
4. 检查长度
5. memcpy 到对象绑定的真实变量
6. 回 download response
```

当前可写对象重点是：

```text
0x2000 LED Output Mask
```

所以 TwinCAT 通过 SDO 写 `0x2000`，实际会改 `g_testCounter`。

### 4.5 SDO Info：TwinCAT CoE Online 能不能显示对象列表的关键

入口：

```c
static void SDO_HandleInfo(uint8_t *rxBuf);
```

它处理：

```text
Get OD List
Get Object Description
Get Entry Description
```

这个不是普通读写对象值，而是 TwinCAT 想知道：

```text
你有哪些对象？
每个对象是什么类型？
每个子项是什么类型、长度、权限？
```

所以 CoE Online 页面能显示出对象字典，主要靠这一块。

## 5. AX58100 和 CoE 的边界

最重要的一条线：

```text
AX58100.c 负责“邮箱字节怎么从 SM0/SM1 进出”
app_coe.c 负责“这些邮箱字节代表什么 SDO 请求”
```

具体调用关系：

```text
CoE_MainTask()
  -> ESC_Mbx_RxFull()
  -> ESC_Mbx_Read()
  -> 解析 Mailbox Header
  -> 解析 CoE Header
  -> 解析 SDO Header
  -> 查找对象字典
  -> 组响应包
  -> ESC_Mbx_TxFull()
  -> ESC_Mbx_Write()
```

过程数据则是另一条线：

```text
ECAT_ProcessDataExchange()
  -> ESC_ReadOutputData()    读 SM2
  -> 应用处理 LED/输入
  -> ESC_WriteInputData()    写 SM3
```

状态机是总开关：

```text
INIT/PREOP：主要配置和邮箱
SAFEOP：过程数据配置好了，但输出还不能正式生效
OP：正式周期性交换过程数据
```

## 7. 几个容易混的概念

### 7.1 Mailbox 和 PDO 不是一回事

```text
Mailbox：
  慢速、命令式、带协议头
  用来跑 CoE/SDO
  对应 SM0/SM1

PDO：
  周期性过程数据
  OP 状态下快速交换
  对应 SM2/SM3
```

### 7.2 CoE Online 读对象，不等于 PDO 在交换

TwinCAT CoE Online 页面读 `0x1008`、`0x1018`，走的是：

```text
邮箱 SM0/SM1 + CoE SDO
```

TwinCAT 周期性控制 LED、读取输入，走的是：

```text
过程数据 SM2/SM3 + FMMU + PDO
```

两条通道有关联，但不是一条通道。

### 7.3 0x2000/0x2001 既可以被 SDO 访问，也可以被 PDO 映射

对象字典里：

```text
0x2000 LED Output Mask
0x2001 Digital Input Mask
```

SDO 访问时：

```text
主站通过 CoE mailbox 单次读写对象
```

PDO 访问时：

```text
对象通过 0x1600/0x1A00 映射到周期过程数据
主站每个周期交换
```

所以同一个对象，可以有两种访问方式：

```text
SDO：调试/配置/偶发访问
PDO：实时周期交换
```

## 9. 一句话总览

```text
AX58100.c：
  把 SPI PDI 封装成 ESC 寄存器、SM、邮箱、过程数据的硬件访问接口。

app_ethercat.c：
  根据 TwinCAT 写入的 AL Control 推进 INIT/PREOP/SAFEOP/OP，并在 OP 下交换过程数据。

app_coe.c：
  从 SM0 邮箱拿 CoE/SDO 请求，查对象字典，然后通过 SM1 回响应。
```
