# AX58100 ESC SPI PDI 通信协议手册

> 基于 AX58100 Full Datasheet v1.03 和 Beckhoff ESC 标准编写
>
> 适用场景：STM32F103 通过 SPI 访问 AX58100 EtherCAT 从站控制器

---

## 一、AX58100 是什么

AX58100 是亚信电子（ASIX）的 2/3 端口 EtherCAT 从站控制器（ESC），内部集成两个百兆 PHY。

```
┌──────────────────────────────────────────────────┐
│                   AX58100                        │
│                                                  │
│  ┌──────┐  ┌──────┐   ┌──────────────────┐      │
│  │ PHY0 │  │ PHY1 │   │   ESC Core        │      │
│  │      │  │      │   │  (Beckhoff IP)    │      │
│  └──┬───┘  └──┬───┘   │                    │      │
│     │         │        │  9KB RAM           │      │
│  RJ45-0   RJ45-1       │  8 FMMU            │      │
│                        │  8 SyncManager     │      │
│                        │  64bit DC Clock    │      │
│                        └────────┬───────────┘      │
│                                 │                  │
│              ┌──────────────────┼───────────────┐  │
│              │   双 SPI 从机接口  │  功能外设      │  │
│              │  SCS_ESC   SCS_FUNC  PWM/DIO/... │  │
│              └──────────────────┴───────────────┘  │
└──────────────────────────────────────────────────┘
```

**两个 SPI 从机接口，同一组 SCK/MOSI/MISO，不同 CS：**

| CS 引脚 | 接口名称 | 用途 |
|---------|---------|------|
| SCS_ESC (79脚) | **PDI SPI** | 访问 ESC 标准寄存器 + 过程数据 RAM |
| SCS_FUNC (80脚) | **Function SPI** | 访问 AX58100 专用外设（PWM/DIO/SPI Master 等） |

当前我们只用了 SCS_ESC 一根 CS，访问 ESC 寄存器。

## 二、硬件接线

```
STM32F103              AX58100
─────────              ───────
PA4 (CS)   ──────────► SCS_ESC (79)
PA5 (SCK)  ──────────► SCLK
PA6 (MISO) ◄────────── MISO
PA7 (MOSI) ──────────► MOSI
GND        ─────────── GND

AX58100 侧:
  SCS_FUNC (80) ──── 上拉到 VDDIO (不用 Function SPI 时必须拉高)
  IRQ         ──── 可选, 接 MCU 外部中断脚
```

## 三、SPI 通信参数

| 参数 | 值 | 说明 |
|------|-----|------|
| **SPI 模式** | Mode 3 (CPOL=1, CPHA=1) | SCK 空闲为高，第二个边沿采样 |
| **时钟频率** | ≤ 20MHz | 当前用 4.5MHz (72/16) |
| **数据位宽** | 8 bit | |
| **位序** | MSB first | 每字节高位先发 |
| **字节序** | Little Endian | 多字节寄存器低字节在前 |
| **CS 极性** | 低有效 (默认) | 可通过 0x0150 配置 |
| **地址模式** | 2 字节 | 寻址 8KB (0x0000~0x1FFF) |

### 为什么是 Mode 3？

数据手册原文："SPI mode 3 is recommended for Slave Sample Code"。

Mode 3 的 SCK 空闲高、第二个边沿采样，对应的 CubeMX 配置是 `CPOL=High`, `CPHA=2Edge`。

## 四、PDI SPI 通信协议

### 4.1 事务结构

每次 SPI 事务分两段，在 CS 为低期间连续完成：

```
CS   ‾‾‾\___________________________________________/‾‾‾
            │      地址段 (2字节)     │   数据段 (N字节)   │
```

### 4.2 地址段格式

地址段占 2 字节，拼入 13 位地址 + 3 位命令：

```
MOSI Byte 0:  A[12:5]                      地址高 8 位
MOSI Byte 1:  A[4:0] << 3 | CMD[2:0]      地址低 5 位 + 命令

MISO Byte 0:  AL Event Request [7:0]       IRQ 状态寄存器 0x0220
MISO Byte 1:  AL Event Request [15:8]      IRQ 状态寄存器 0x0221
```

**编址举例：**

```
读寄存器 0x0000:
  addr = 0x0000
  A[12:0] = 0 0000 0000 0000
  Byte 0 = (0x0000 >> 5) & 0xFF = 0x00
  Byte 1 = ((0x0000 & 0x1F) << 3) | 0x03 = 0x03
  → MOSI 发 [0x00, 0x03, ...]

读寄存器 0x0140:
  addr = 0x0140
  A[12:0] = 0 0001 0100 0000
  Byte 0 = (0x0140 >> 5) & 0xFF = 0x0A
  Byte 1 = ((0x0140 & 0x1F) << 3) | 0x03 = 0x03
  → MOSI 发 [0x0A, 0x03, ...]

读寄存器 0x1000:
  addr = 0x1000
  A[12:0] = 1 0000 0000 0000
  Byte 0 = (0x1000 >> 5) & 0xFF = 0x80
  Byte 1 = ((0x1000 & 0x1F) << 3) | 0x03 = 0x03
  → MOSI 发 [0x80, 0x03, ...]
```

### 4.3 命令码 (CMD[2:0])

| CMD | 二进制 | 含义 | 使用场景 |
|-----|--------|------|---------|
| 0 | 000 | NOP | 无操作 |
| 2 | 010 | **读**（无等待状态） | **块读用这个** |
| 3 | 011 | 读 + 等待状态字节 | 单字节读用这个 |
| 4 | 100 | **写** | 所有写操作 |
| 6 | 110 | 地址扩展 | 3 字节地址模式 |
| 1/5/7 | — | 保留 | — |

### 4.4 单字节读时序（CMD=0x03，含等待字节）

```
MOSI: [A[12:5]] [A[4:0]|CMD=011] [0xFF 等待] [0xFF 读数据]
MISO: [  IRQ0  ] [    IRQ1      ] [   xx     ] [   DATA   ]

字节 0       字节 1          字节 2        字节 3
└─地址段─┘                └──── 数据段 ────┘
                               ↑ 等待字节：给 ESC 240ns 取数据
                                          ↑ 读 1 字节时，唯一的那个字节一定是 0xFF（终止）
```

### 4.5 块读时序（CMD=0x02，无等待字节） ⭐ 关键

```
MOSI: [A[12:5]] [A[4:0]|CMD=010] [0x00] ... [0x00] [0xFF]
MISO: [  IRQ0  ] [    IRQ1      ] [d0  ] ... [dN-2] [dN-1]

字节 0       字节 1          字节 2       字节 N+1   ← 共 2+N 字节
└─地址段─┘              └──────── 数据段 ──────────┘
                            ↑ 3 个 NOP 替代 wait byte
                            非终止字节: MOSI=0x00 → ESC 继续预取
                                               最后一个: MOSI=0xFF → 读终止
```

**为什么块读不能所有数据字节都是 0xFF？**

> 手册原文：*"If MOSI is low during a data byte transfer, at least one more byte will be read by the master afterwards."*

| MOSI | ESC 行为 |
|------|---------|
| ≠ 0xFF | "后面还有数据，继续预取" |
| = 0xFF | "这是最后一字节，停止" |

如果从第一个数据字节就用 0xFF，ESC 读 1 个字节就停了，后面收到的全是垃圾。这就是之前 `g_escError=7` 的根因。

```
正确（4 字节块读）:
  MOSI: [addr0] [addr1] [0x00] [0x00] [0x00] [0xFF]
                          ↑      ↑      ↑      ↑
                         继续   继续   继续   终止

错误（旧版 bug）:
  MOSI: [addr0] [addr1] [0xFF] [0xFF] [0xFF] [0xFF]
                          ↑
                     ESC: "终止！后面不读了！"
                          后面 3 个字节全是 PDI Error
```

### 4.6 写操作时序（CMD=0x04）

```
MOSI: [A[12:5]] [A[4:0]|CMD=100] [数据0] [数据1] ...
MISO: [  IRQ0  ] [    IRQ1      ] [  xx  ] [  xx  ] ...

字节 0       字节 1          字节 2+
└─地址段─┘              └── 数据段 ──┘
                         MISO 在此期间无意义
```

写操作不需要等待字节，地址段后面直接跟数据。

### 4.5 写操作时序（CMD=0x04）

```
MOSI: [A[12:5]] [A[4:0]|CMD=100] [数据0] [数据1] ...
MISO: [  IRQ0  ] [    IRQ1      ] [  xx  ] [  xx  ] ...

字节 0       字节 1          字节 2+
└─地址段─┘              └── 数据段 ──┘
                         MISO 在此期间无意义
```

写操作不需要等待字节，地址段后面直接跟数据。

## 五、ESC 寄存器速查

### 5.1 核心寄存器

| 地址 | 寄存器名 | 说明 |
|------|---------|------|
| **0x0000** | ESC Type | ESC 芯片类型（AX58100 返回值取决于型号） |
| **0x0001** | ESC Revision | 芯片版本号 |
| 0x0004 | FMMU Count | FMMU 数量 |
| 0x0005 | SyncMan Count | SyncManager 数量 |
| 0x0008 | RAM Size | RAM 容量（KB） |
| 0x0010~0x0017 | Station MAC | 站 MAC 地址 |

### 5.2 PDI 控制寄存器

| 地址 | 寄存器名 | 说明 |
|------|---------|------|
| **0x0140** | PDI Control | PDI 类型选择。**必须 = 0x05 才是 SPI Slave** |
| **0x0150** | PDI Configuration | SPI 模式配置（位[1:0] = 11 即 Mode 3） |
| 0x0151 | Sync/Latch Config | DC 同步信号配置 |

```
0x0140 的值由 EEPROM 加载，不能通过 SPI 修改。
如果 EEPROM 未烧录或 PDI 类型不是 0x05，SPI 无法通信！
```

### 5.3 中断状态寄存器

| 地址 | 寄存器名 | 说明 |
|------|---------|------|
| 0x0220 | AL Event Request [7:0] | 中断请求状态低 8 位 |
| 0x0221 | AL Event Request [15:8] | 中断请求状态高 8 位 |
| 0x0222 | AL Event Request [23:16] | 3 字节地址模式才有 |

这几个寄存器**不需要主动读**。每次 SPI 事务的地址段，AX58100 自动把它们送到 MISO 上。代码里存到 `g_esc_irq0/1`，随时可查。

### 5.4 看门狗

| 地址 | 寄存器名 | 说明 |
|------|---------|------|
| 0x0400 | Watchdog Divider | 看门狗时钟分频 |
| 0x0410 | PDI Watchdog Time | PDI 看门狗超时 |

### 5.5 SyncManager 寄存器

| 地址范围 | 说明 |
|---------|------|
| 0x0800~0x080F | SM0 配置（邮箱输出） |
| 0x0810~0x081F | SM1 配置（邮箱输入） |
| 0x0820~0x082F | SM2 配置（过程数据输出） |
| 0x0830~0x083F | SM3 配置（过程数据输入） |

每个 SM 占 16 字节，偏移 2 为物理起始地址（`+0x2`），偏移 6 为长度（`+0x6`）。

### 5.6 过程数据 RAM

| 地址 | 说明 |
|------|------|
| **0x1000~0x1FFF** | 用户过程数据 RAM（4KB 起始，AX58100 实际有 9KB） |

这是 PDI 侧可以自由读写的 RAM 区域，EtherCAT 主站也可以通过 FMMU/SyncManager 访问。

### 5.7 错误诊断

| 地址 | 寄存器名 | 说明 |
|------|---------|------|
| 0x030D | PDI Error Counter | SPI 访问错误计数 |
| 0x030E | PDI Error Code | 最后一次错误的原因码 |

Error Code 常见值：`4` = 缺少读终止字节，`5` = 终止后继续读。

## 六、上电启动流程

```
上电 → 内部复位 → 检测 EEPROM → 加载 ESI 配置 → PDI 就绪
                    │
                    └── ESC Configuration Area (14 bytes)
                        决定了:
                        · PDI 类型 (必须是 0x05 = SPI Slave)
                        · SPI 模式
                        · PHY 配置
                        · 其他硬件配置
```

**关键：如果 EEPROM 没有正确烧录，AX58100 不会启用 SPI PDI，所有 SPI 通信无响应。**

## 七、软件接口速查

### 单寄存器读写

```c
uint8_t data;

// 读 0x0000 (ESC Type)
ESC_ReadRegister(0x0000, &data);

// 写 0x0F00 (User RAM)
ESC_WriteRegister(0x0F00, 0xA5);
```

### 块读写

```c
uint8_t buf[32];

// 从 0x1000 读 32 字节
ESC_ReadBlock(0x1000, buf, 32);

// 向 0x1000 写 32 字节
ESC_WriteBlock(0x1000, buf, 32);
```

### 获取中断状态

```c
uint8_t irq0, irq1;
ESC_GetIRQStatus(&irq0, &irq1);
// irq0: AL Event Request [7:0]  (0x0220)
// irq1: AL Event Request [15:8] (0x0221)
```

### 单字节底层收发（不走 ESC 协议）

```c
uint8_t rx;
SPI_WriteByte(0x55);              // 只发
SPI_ReadByte(&rx);                // 只收 (发 0xFF 提供时钟)
SPI_WriteReadByte(0xAA, &rx);     // 同时收发
```

## 八、常见 SPI 访问错误

| 错误现象 | 可能原因 |
|---------|---------|
| 通信完全无响应 | EEPROM 未烧录 / PDI 类型不是 0x05 |
| PDI Error Counter 持续递增 | 缺少读等待字节 / 终止字节不是 0xFF |
| 字节错位 | CS 中间被释放 / 字节数不是 8 的倍数 |
| 读到全 0 或 全 FF | SPI Mode 不对 (应该 Mode 3) / SCK 频率超限 |

### 调试建议

1. **先用示波器抓 SCK 和 MOSI**，确认波形、时钟极性、相位正确
2. **查看 PDI Error Counter (0x030D)**，如果不为 0 说明有协议错误
3. **先读 0x0000 和 0x0001**，这两个寄存器永远可读，用来确认通信基本正常
4. **读 0x030E** 看具体错误原因

## 九、事务示例

### 9.1 单字节读 0x0000（CMD=0x03，ECS_ReadRegister）

```
目标: 读 ESC Type 寄存器 (0x0000)

步骤：
1. 编码地址:
   addr0 = (0x0000 >> 5) & 0xFF = 0x00
   addr1 = ((0x0000 & 0x1F) << 3) | 0x03 = 0x03

2. 构建发送缓冲区 (4 字节):
   txBuf = [0x00, 0x03, 0xFF, 0xFF]
           地址0  地址1  等待  数据+终止

3. CS 拉低，收发 4 字节:
   ┌──────┬──────┬──────┬──────┐
   │ MOSI │ 0x00 │ 0x03 │ 0xFF │ 0xFF │
   ├──────┼──────┼──────┼──────┼──────┤
   │ MISO │ IRQ0 │ IRQ1 │  xx  │ 0xC8 │  ← ESC Type
   └──────┴──────┴──────┴──────┘

4. 等 BSY=0，CS 拉高
5. 返回 rxBuf[3] = 0xC8
```

### 9.2 块读 8 字节 0x1000（CMD=0x02，ECS_ReadBlock）

```
目标: 从用户 RAM 0x1000 读 8 字节

1. 编码: addr0=0x80, addr1=(0x00<<3)|0x02=0x02

2. 构建发送缓冲区 (2+8=10 字节):
   txBuf = [0x80, 0x02, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0xFF]
           地址0  地址1  ←── 7 字节 0x00 (继续预取) ──→ 终止

3. CS 拉低，NOP×3(≥240ns)，收发 10 字节

4. 等 BSY=0，CS 拉高

5. rxBuf[2..9] = 8 字节数据
```

## 十、实际调试验证记录 & 踩坑总结

> 以下是本次 SPI 驱动开发过程中实际遇到的问题和排查过程。

### 验证通过清单

```
✅ ESC Type     = 0xC8  (ASIX AX58100)
✅ ESC Version  = 0x00  (Rev.0)
✅ testRd       = 0xA5  (0x1F00 单字节读写)
✅ 0x1000 块读写 = {0xA5,0x5A,0x01,0x02,0x03,0x04,0x55,0xAA} (8字节完全匹配)
✅ SM0 配置     = {0x00,0x00,0x00,0x00,0x00,0x30,0x00,0x01} (起始地址=0,长度=0,未启用)
✅ pdiErrCnt    = 0     (断电重启后清零)
```

### Bug #1 — 块读所有数据字节都是 0xFF 🔴

| 项目 | 内容 |
|------|------|
| **现象** | 单字节读正常(0xC8)，块读 8 字节有 7 个错误 `g_escError=7`，`pdiErrCnt=0xFF`, `pdiErrCode=0x60` |
| **根因** | AX58100 数据手册规定数据字节中**只有最后一个可以是 0xFF（读终止信号）**。0xFF 告诉 ESC "停止预取"。块读代码把所有字节都设成了 0xFF，ESC 读完第一个就停了 |
| **修复** | 前 N-1 数据字节 = `0x00`，第 N 字节 = `0xFF` |
| **教训** | 0xFF 在 ESC SPI 协议里是读终止命令，不是普通的 dummy clock！ |

```
错误: [addr0][addr1][0xFF][0xFF][0xFF][0xFF]  ← 第1个数据字节 ESC 就停机
正确: [addr0][addr1][0x00][0x00][0x00][0xFF]  ← 只有最后一个才是 0xFF
                          继续   继续   终止
```

### Bug #2 — CMD=0x03 在块读时语义冲突 🔴

| 项目 | 内容 |
|------|------|
| **现象** | 修了 Bug#1 后块读仍不正常 |
| **根因** | CMD=0x03 协议中地址段后有一个 `0xFF` wait state 字节，和后续数据字节的 0xFF 语义冲突。CMD=0x02 不需要 wait state 字节，协议更简洁 |
| **修复** | 块读改用 `CMD=0x02`，用 `__NOP()×3` 替代 wait byte，满足 t_read >= 240ns |

| 读模式 | CMD | 字节数 | 数据起始于 RX | 适用场景 |
|--------|-----|--------|-------------|---------|
| 带等待 | 0x03 | 3+N | rxBuf[3] | 单字节读 (ESC_ReadRegister) |
| 无等待 | 0x02 | 2+N | rxBuf[2] | 块读 (ESC_ReadBlock) |

### Bug #3 — CS 提前释放 🟡

| 项目 | 内容 |
|------|------|
| **现象** | 偶尔零星 PDI Error |
| **根因** | HAL_SPI_TransmitReceive 返回时 SPI 移位寄存器可能还在发送最后一个 bit |
| **修复** | SPI_CS_HIGH 宏中先 `while(SPI_FLAG_BSY){}` 等待硬件空闲 |

### Bug #4 — pdiErrCnt 是只增不减的累积值 🟡

| 项目 | 内容 |
|------|------|
| **现象** | 修完所有 Bug 后 pdiErrCnt 仍显示 0xFF，误以为没修好 |
| **根因** | PDI Error Counter (0x030D) 只增不减。之前跑 Bug 代码积累的 255 个错误一直保留 |
| **教训** | 调试前先给 AX58100 断电重启清零，或直接写 0 到 0x030D |

### Bug #5 — CS 上电初始为低 🟡

| 项目 | 内容 |
|------|------|
| **现象** | 上电瞬间 PA4=0（CubeMX 默认 Output Low），AX58100 被意外选中 |
| **修复** | MX_SPI1_Init 末尾调用 SPI_CS_HIGH() |

### Bug #6 — 块读写函数栈溢出风险 🟡

| 项目 | 内容 |
|------|------|
| **现象** | 无实际崩溃，但栈上分配 txBuf[259]+rxBuf[259]=518 字节，接近默认栈上限 |
| **修复** | 改为 static 缓冲区（放 BSS 段），ESC_MAX_BLOCK_SIZE 从 256 降至 128 |

### 最终正确协议速查

```
单字节读 (CMD=0x03):  [addr0][addr1|0x03][0xFF wait][0xFF data+term]  = 4 字节
块读     (CMD=0x02):  [addr0][addr1|0x02][0x00 x N-1][0xFF term]  = 2+N 字节
                       NOPx3 延时 >=240ns
写       (CMD=0x04):  [addr0][addr1|0x04][data...]  = 2+N 字节

CS 时序: LOW -> 收发 -> while(BSY){} -> HIGH
```
