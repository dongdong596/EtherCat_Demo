## 架构拆分

### 拆分前

```
Core/spi.c ── 所有代码混在一起
  ├── SPI 硬件配置 (CubeMX)
  ├── 通用 SPI 收发
  ├── ESC PDI 协议 (地址编码/CMD=0x02/0x03/0x04)
  ├── ESC 寄存器读写
  ├── ESC 测试/诊断
  └── (无处安放的状态机)
```

### 拆分后

```
App/app_ethercat.c   ←  EtherCAT 应用层 (状态机 / CoE / 过程数据)
BSP/AX58100.c        ←  ESC 驱动 (PDI 协议 + 寄存器读写 + 地址映射)
Core/spi.c           ←  SPI 硬件抽象 (CS 控制 + 字节收发)
```

| 层 | 职责 | 依赖 |
|----|------|------|
| `Core/spi.c` | SPI 硬件：CS 控制、单/多字节收发、回环测试 | HAL |
| `BSP/AX58100.c` | ESC 驱动：地址编码、CMD=0x02/03/04、寄存器/块读写、IRQ 捕获 | spi.h |
| `App/app_ethercat.c` | EtherCAT 协议：状态机、CoE、过程数据 | AX58100.h |

**设计原则**：每层只依赖下面一层，上层不碰下层细节。比如 app_ethercat.c 不调 HAL_SPI_TransmitReceive，只调 ESC_ReadRegister / ESC_WriteRegister。

## 状态机实现

### EtherCAT 状态跳转图

```
(上电) → Init ⇄ PreOp ⇄ SafeOp ⇄ Op
            ↑        ↑        ↑
         可配寄存器  可邮箱    可过程数据
```

### 协议流程

```
主站 (TwinCAT)                   从站 (STM32 + AX58100)
     │                                    │
     │  网线: "切到 PreOp"                 │
     │  写 AL Control (0x0120) = 0x02     │
     │ ─────────────────────────────────► │
     │                                    │ ECAT_MainTask() 读 0x0120
     │                                    │ IsValidTransition(Init→PreOp) ✓
     │                                    │ OnLeaveState(Init)
     │                                    │ OnEnterState(PreOp)
     │                                    │ 写 AL Status (0x0130) = 0x02
     │                                    │
     │  网线: 读 AL Status                 │
     │ ◄───────────────────────────────── │
     │  确认: AL Status = 0x02             │
     │  下一步: 配 SyncManager             │
```

### AL Control 和 AL Status 到底是什么

就是 ESC 硬件里的**两个寄存器**，只靠这一对寄存器完成全部握手：

| 寄存器 | 地址 | 谁写 | 谁读 | 比喻 |
|--------|------|------|------|------|
| AL Control | 0x0120 | 主站（网线） | 从站（SPI） | 工头的指令单 |
| AL Status | 0x0130 | 从站（SPI） | 主站（网线） | 工人的回复单 |
| AL Status Code | 0x0134 | 从站（SPI） | 主站（网线） | 出错时的原因 |

**不是专门给 PreOp 用的**，四个状态切换都走这对寄存器，只是值不同：

```
AL Control / AL Status 的 bit[3:0]:
  0x01 = Init       "啥都别干，刚上电"
  0x02 = PreOp      "准备配数据通道"
  0x04 = SafeOp     "验证配置，数据只读"
  0x08 = Op         "开始干活，全速运行"
```

全程只变这一对寄存器里的这一个字段：

```
         主站写 AL Control         从站写 AL Status
         ────────────────          ────────────────
Init     (上电默认 0x01)
  ↓         写 0x02   ──────────►     回 0x02
PreOp
  ↓         写 0x04   ──────────►     回 0x04
SafeOp
  ↓         写 0x08   ──────────►     回 0x08
Op
```

一对寄存器，四个不同的值，四次握手。这就是 EtherCAT 状态机的全部。

### 关键函数

```c
// 每 10ms 调一次
uint8_t ECAT_MainTask(void)
{
    读 AL Control (0x0120);     // 主站想让我切到哪个状态
    return _ECAT_DoTransition(requestedState);  // 验证 + 执行 + 写 AL Status
}

// 核心跳转逻辑 (ECAT_MainTask 和 ECAT_SelfTest 共用)
static uint8_t _ECAT_DoTransition(uint8_t requestedState)
{
    if (requestedState == current)    return;      // 没变, 不处理
    if (!IsValidTransition(from, to)) return;      // 非法跳转, 写错误码
    OnLeaveState(old);                             // 离开旧状态
    OnEnterState(new);                             // 进入新状态 (占位, 第5~7步填)
    写 AL Status;                                  // 回应主站
}
```

## 踩坑：AL Control 不能用 PDI 写

### 问题

自测函数尝试通过 SPI 写 AL Control (0x0120) 来模拟主站请求，返回 `0x1F`（5 个跳转全失败）。

### 原因

| 寄存器 | 谁写 | PDI 能写? |
|--------|------|-----------|
| AL Control (0x0120) | **主站**通过网线 | ❌ PDI 只读 |
| AL Status (0x0130) | **从站**通过 PDI | ✅ |
| AL Status Code (0x0134) | **从站**通过 PDI | ✅ |

这是 Beckhoff ESC 标准的安全设计——如果 PDI 能写 AL Control，STM32 越权改状态，主站就懵了。

### 修复

把核心跳转逻辑抽成 `_ECAT_DoTransition()`：

```
ECAT_MainTask:    读 AL Control → _ECAT_DoTransition(state)   // 上线用
ECAT_SelfTest:    直接调 _ECAT_DoTransition(state)             // 自测用
```

自测绕开 AL Control 寄存器，直接调同一套跳转逻辑，既能验证状态机正确性，又不污染生产代码。

## 自测验证

`ECAT_SelfTest()` 手动跑一遍完整状态机流程：

```
Init → PreOp → SafeOp → Op → SafeOp → PreOp → Init
  │       │        │      │       │         │       │
  └───────┴────────┴──────┴───────┴─────────┴───────┘
                 6 次跳转全部验证
```

返回值 0x00 = 全部通过。每步同时检查软件状态 `ECAT_GetState()` 和硬件 AL Status 寄存器，双重确认。
