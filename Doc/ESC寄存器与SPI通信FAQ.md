# ESC 寄存器与 SPI 通信 FAQ

> 基于 AX58100 + STM32F103 实际开发过程中的问答整理

---

## 一、单寄存器读 vs 块读的区别

### 协议层面

| | 单寄存器读 | 块读 |
|---|---|---|
| **函数** | `ESC_ReadRegister(addr, &data)` | `ESC_ReadBlock(addr, buf, size)` |
| **CMD** | `0x03` (Read + Wait State) | `0x02` (Read, 无等待) |
| **事务长度** | 4 字节 | 2 + N 字节 |
| **数据起始于** | rxBuf[3] | rxBuf[2] |

### 单字节读时序 (CMD=0x03)

```
MOSI: [addr0] [addr1|0x03] [0xFF 等待] [0xFF 数据+终止]
MISO: [ IRQ0 ] [  IRQ1    ] [   xx    ] [   数据     ]

字节 0     字节 1         字节 2       字节 3
└─地址段─┘              └─── 数据段 ────┘
```

地址段后有 **1 个 wait state 字节 (0xFF)**，给 ESC 内部 ≥240ns 取数据时间。

### 块读时序 (CMD=0x02)

块读拆为**两次 SPI 传输**，CS 全程拉低，NOP 延时放在两次传输之间：

```
【第一次传输: 地址段 2 字节】
MOSI: [addr0] [addr1|0x02]
MISO: [ IRQ0 ] [  IRQ1    ]

  → NOP×10 (满足 ESC t_read ≥ 240ns)

【第二次传输: 数据段 N 字节】
MOSI: [0x00] ... [0x00] [0xFF]
MISO: [d0  ] ... [dN-2] [dN-1]
                      ↑ 前 N-1 个 = 0x00 (继续预取)
                                  ↑ 最后一个 = 0xFF (终止)
```

**没有 wait state 字节**，改为拆成两次 HAL 调用，在地址和数据之间插入 NOP 延时，满足 t_read ≥ 240ns。
### 为什么块读不能直接用 CMD=0x03？

CMD=0x03 的 wait state 字节是 0xFF，数据终止字节也是 0xFF。单字节读时两者不冲突（一个是第 3 字节、一个是第 4 字节），但块读时多个 0xFF 连续出现，ESC 读到第一个就停了。

CMD=0x02 去掉 wait state 字节，数据段里只有最后一个字节是 0xFF，消除了歧义。

### 关键规则

> **MOSI ≠ 0xFF 时 ESC 认为"后面还有数据"，MOSI = 0xFF 时 ESC 停止预取。**

## 二、什么时候用单字节读，什么时候用块读

| | 单字节读 | 块读 |
|---|---|---|
| **判据** | 地址**分散** | 地址**连续** |
| **典型目标** | Type, Version, PDI Error, AL Control 等零散寄存器 | SM 配置, FMMU 配置, 过程数据 RAM |
| **数据量** | 1 字节 | 2~128 字节 |
| **效率** | 每次 4 字节 SPI 传输 | 一次 2+N 字节，快 3~4 倍 |

```
读 16 字节 SM0 配置:

  方案A (单字节 ×16): 64 字节 SPI, 16 次 CS/BSY
  方案B (块读 ×1):    18 字节 SPI, 1 次 CS/BSY
```

## 三、地址编码函数

```c
static void ESC_EncodeAddress(uint16_t addr, uint8_t cmd,
                               uint8_t *pAddr0, uint8_t *pAddr1)
{
    *pAddr0 = (uint8_t)((addr >> 5) & 0xFF);            // A[12:5]
    *pAddr1 = (uint8_t)(((addr & 0x1F) << 3) | (cmd & 0x07)); // A[4:0]<<3 | CMD
}
```

把 13 位地址 + 3 位命令拼成 SPI 地址段的 2 个字节：

```
地址段结构 (2 字节):

  Byte 0:  A[12] A[11] A[10] A[9] A[8] A[7] A[6] A[5]   ← 地址高 8 位
  Byte 1:  A[4]  A[3]  A[2]  A[1]  A[0]  CMD2 CMD1 CMD0  ← 地址低 5 位 + 命令 3 位
```

**为什么是 13 位地址？** 13 位 = 8192 字节 (0x0000~0x1FFF)，刚好覆盖 8KB 地址空间。

### 编码举例

```c
// 读 0x0140
addr = 0x0140 = 0 0001 0100 0000
a0 = (0x0140 >> 5) = 0x0A
a1 = ((0x00) << 3) | 0x03 = 0x03
// → MOSI [0x0A, 0x03, ...]

// 读 0x1000
addr = 0x1000 = 1 0000 0000 0000
a0 = 0x80
a1 = 0x02
// → MOSI [0x80, 0x02, ...]
```

## 四、ESC 寄存器映射（固定，Beckhoff 标准）

所有符合 ESC 标准的芯片（AX58100、ET1100、LAN9252 等）都遵循同一套布局。

```
0x0000 ────────────────────────
  0x0000~0x000F   身份信息       Type, Revision, FMMU/SM 数量, RAM 大小
  0x0010~0x0017   MAC 地址       站地址
  0x0100~0x011F   状态机控制      AL Control, AL Status, AL Status Code
  0x0120~0x013F   
  0x0140~0x015F   PDI 配置        PDI 类型, SPI 模式, CS 极性
  0x0200~0x03FF   中断/状态       AL Event Request, PDI Error, Watchdog
  0x0400~0x04FF   看门狗       
  0x0600~0x06FF   FMMU 配置       8 个 FMMU, 各 16 字节
  0x0800~0x087F   SM 配置         8 个 SyncManager, 各 16 字节
  0x0900~0x09FF   分布式时钟      DC 同步配置
  0x0E00~0x0FFF   EEPROM/其他  
0x1000 ────────────────────────
  0x1000~0x1FFF   过程数据 RAM    你自由读写的地方
                  (AX58100 实际 9KB)
```

## 五、SyncManager（SM）— 数据通道管理员

SM 是 ESC 里最重要的配置，管一件事：**RAM 里哪块区域归哪个通道，数据流向是什么方向**。

### SM 结构（每 SM 16 字节，共 8 个）

以 SM2 为例，基址 = 0x0800 + 2×16 = 0x0820：

```
地址      偏移  内容                  比喻
──────────────────────────────────────────────
0x0820    0-1   物理起始地址 = 0x1200  "从第 1200 号格子开始"
0x0822    2-3   长度       = 0x0020  "占 32 个格子"
0x0824    4     控制寄存器            "收件柜还是发件柜"
0x0825    5     状态寄存器 (自动)     "柜子里有快递吗"
0x0826    6     激活       = 0x01    "启用"
0x0827    7     PDI 控制             "STM32 这边怎么用"
```

### 8 个 SM 的约定用法

| SM | 用途 | 方向 |
|----|------|------|
| SM0 | 邮箱输出（主站→STM32） | 主站写，STM32 读 |
| SM1 | 邮箱输入（STM32→主站） | STM32 写，主站读 |
| SM2 | 过程输出（主站→STM32） | 主站写，STM32 读 |
| SM3 | 过程输入（STM32→主站） | STM32 写，主站读 |
| SM4~7 | 备用 | 按需分配 |

### 控制寄存器（偏移 4）

```
Bit 7-4:  保留
Bit 3-1:  模式
            000 = 缓冲模式（过程数据 SM2/SM3）
            010 = 邮箱模式（SM0/SM1）
Bit 0:    方向
            0 = 主站写，STM32 读
            1 = STM32 写，主站读
```

### 缓冲模式 vs 邮箱模式

```
缓冲模式（过程数据）:
  新数据直接覆盖旧数据，不关心你有没有取走
  适用于: IO 状态，只关心最新值

邮箱模式（配置命令）:
  写完后置"有新数据"标记，读走后清除
  标记未清则不能写新数据
  适用于: 命令/响应，一条不能丢
```

### 运转流程

```
主站配好 SM2 (起始 0x1200, 长度 32, 方向=主站→STM32) 后:

1. 主站帧到达 → ESC 硬件把数据写入 0x1200
2. SM2 状态寄存器自动更新 → IRQ 位置 1
3. 下一次 SPI 事务时 MISO 上自动带回 IRQ 变化
4. STM32 调 ESC_ReadBlock(0x1200, buf, 32) 取走数据

全程 STM32 不需要轮询，ESC 硬件自动完成步骤 1~3。
```

### 配置示例

```c
// 配 SM2: 主站→STM32, 缓冲模式, 起始 0x1200, 长度 32
ESC_WriteRegister(0x0820, 0x00);  // 起始地址低字节
ESC_WriteRegister(0x0821, 0x12);  // 起始地址高字节
ESC_WriteRegister(0x0822, 0x20);  // 长度低字节 = 32
ESC_WriteRegister(0x0823, 0x00);  // 长度高字节 = 0
ESC_WriteRegister(0x0824, 0x00);  // 控制: 缓冲模式, 方向=主站→STM32
ESC_WriteRegister(0x0826, 0x01);  // 激活
```

## 六、FMMU — 地址映射

FMMU 做**地址翻译**：主站用逻辑地址访问从站，FMMU 把逻辑地址映射到物理 RAM。

```
主站: "把 2 字节写到逻辑地址 0x0000"
  │
  ▼ FMMU0 映射: 逻辑 0x0000~0x0001 → 物理 0x1200~0x1201
  │
  ▼ 物理 RAM 0x1200 被写入
  │
  ▼ STM32 从 0x1200 读到数据
```

### SM 和 FMMU 的关系

```
SM 管:  "RAM 哪块区域属于哪个通道"
FMMU 管: "主站的逻辑地址对应哪块物理 RAM"

两者配合:
  SM2 说:   "0x1200~0x12FF 是过程数据输出"
  FMMU0 说: "逻辑 0x0000 → 物理 0x1200"

结果: 主站写逻辑 0x0000 → 自动到 0x1200 → STM32 从 0x1200 读
```

## 七、中断/状态寄存器（0x0200~0x03FF）

硬件自动更新，你只读不写。

| 地址 | 寄存器 | 说明 |
|------|--------|------|
| 0x0220~0x0221 | AL Event Request | SM 数据更新、状态切换、看门狗触发等事件标记 |
| 0x030D | PDI Error Counter | SPI 出错次数（只增不减，需断电清零） |
| 0x030E | PDI Error Code | 最后一次错误的原因码 |

### AL Event Request 不需要主动读

每次 SPI 事务的地址段，AX58100 自动把 0x0220/0x0221 的值送上 MISO，代码自动存在 `g_esc_irq0/1`：

```c
ESC_ReadRegister(0x0000, &data);  // 读任意寄存器后
// g_esc_irq0 已经是最新中断状态，无需专门读 0x0220
```

## 八、三者的协作关系

```
主站发过程数据（每周期自动发生）:

1. 主站以太网帧到达 AX58100
2. FMMU 翻译: 逻辑地址 → 物理 RAM 地址
3. ESC 硬件写入物理 RAM（SM 管理区域）
4. SM 检测到写入，更新 IRQ 状态（0x0220）
5. 下一次 SPI 事务时 MISO 自动带回 IRQ 变化
6. STM32 用块读取数据走

STM32 的代码只做了第 6 步，其余全是 ESC 硬件自动完成。
```

---

## 九、核心概念一句话

> **AX58100 是一块"共享内存板"**。主站通过网线往里写/往外读，你通过 SPI 往里写/往外读。你的代码不直接和 EtherCAT 协议打交道——你只需在正确的地址写入正确的值，AX58100 自动完成以太网帧的拼包拆包。你要做的事：**配好 SM 地址 → 响应状态机 → 在指定 RAM 位置读写过程数据 → 写 ESI 文件告诉主站你的布局。**
